# Database migrations

Schema changes for the API Gateway's database are managed with
[Alembic](https://alembic.sqlalchemy.org/), scoped to the schemas it currently
owns: `auth`, `budget`, and `transactions` (see
[`docs/architecture.md` § Database strategy](architecture.md#database-strategy)).
As services split out of the monolith, each one gets its own Alembic setup
scoped to its own schema — this is the first one.

All commands below are run from `services/api-gateway`, with the virtualenv
active.

## Prerequisites

- Postgres running: `docker compose up -d postgres` (from the repo root).
- Python deps installed, including `alembic` (already in `requirements.txt`):
  ```bash
  cd services/api-gateway
  python -m venv .venv
  .venv/Scripts/activate        # Windows; use `source .venv/bin/activate` on macOS/Linux
  pip install -r requirements.txt
  ```
- A repo-root `.env` (copy `.env.example` if you don't have one). Alembic
  reads the same `POSTGRES_*` settings as the app itself
  (`app/core/config.py`), and — like the pytest suite — connects from the
  host, so it overrides `POSTGRES_HOST` to `localhost` regardless of what's
  in `.env` (`postgres`, the docker-compose service name, only resolves
  from inside the compose network).

## Running migrations

Apply all pending migrations:

```bash
cd services/api-gateway
alembic upgrade head
```

This creates the `auth`, `budget`, and `transactions` schemas (if they don't
already exist — normally `infra/postgres/init/01_create_schemas.sql` creates
them the first time the postgres container's volume is initialized) and every
table currently defined under `app/db/models/`.

Other useful commands:

```bash
alembic current              # show the currently applied revision
alembic history               # list all revisions
alembic downgrade -1          # roll back one revision
alembic downgrade base        # roll back everything
```

### Seed data (optional)

`infra/postgres/seed/seed_data.sql` has sample rows for local/manual testing
against the running API. It's not part of the automatic docker-entrypoint
init (those scripts only run once, before any tables exist) — run it
yourself after migrating:

```bash
docker compose exec -T postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" < infra/postgres/seed/seed_data.sql
```

## Creating a new migration

1. Change the SQLAlchemy models under `app/db/models/`.
2. Autogenerate a revision from the model diff:
   ```bash
   alembic revision --autogenerate -m "short description"
   ```
3. Open the generated file in `migrations/versions/` and check it — autogenerate
   is a starting point, not the final answer. It won't detect everything
   (e.g. changes to `CHECK` constraints or plain-string enum values), so
   compare it against your model change before committing.
4. Apply it locally (`alembic upgrade head`) and confirm it does what you
   expect before opening a PR.

## How it's wired up

- `alembic.ini` — Alembic config. No `sqlalchemy.url` is hardcoded here;
  `migrations/env.py` builds the connection URL from `Settings` at runtime,
  the same way `app/db/session.py` does.
- `migrations/env.py` — uses the app's async engine (`asyncpg`, matching
  `app/db/session.py`) and points `target_metadata` at `Base.metadata` from
  `app/db/models`, so every model registered there is in scope for
  autogenerate.
- `migrations/versions/` — one file per revision, plain Python + SQLAlchemy
  `op.*` calls generated by Alembic.
- The `alembic_version` tracking table lives in the `budget` schema rather
  than `public` — keeping each service's migration state inside its own
  schema avoids collisions if/when another service's Alembic setup runs
  against the same Postgres instance.
